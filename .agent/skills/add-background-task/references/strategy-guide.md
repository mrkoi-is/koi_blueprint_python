# Background Task Strategy Guide

## 何时选哪种方案

| 方案 | 适用场景 | 不适合场景 |
|---|---|---|
| `FastAPI BackgroundTasks` | 请求完成后顺手做一个短任务；失败可接受；无需重试与持久化 | 需要跨进程、延迟任务、定时任务、结果追踪 |
| `ARQ` | 已有 Redis；想保留 async 技术栈；任务复杂度中等 | 需要复杂 canvas / chord / 大规模生态能力 |
| `Celery` | 需要成熟的 worker 生态、定时任务、重试、独立扩缩容 | 只是一个简单的“发邮件”小任务 |

## 决策信号

优先看以下问题：

1. 任务是否必须在服务重启后继续存在？
2. 是否需要自动重试、延迟、定时调度？
3. 是否需要把任务与 Web 进程分开部署？
4. 是否已经接受 Redis / RabbitMQ 作为运行时依赖？
5. 是否需要更严格的任务监控、死信、告警与运营能力？

## 推荐默认值

- 默认项目：先用同步实现；只有在确有价值时再引入后台任务。
- 最简单异步：先选 `FastAPI BackgroundTasks`。
- Redis 已是基础设施，且想维持 async 端到端：优先 `ARQ`。
- 企业级任务编排：优先 `Celery`。

## 统一约束

- task 名称、队列名称、重试次数要进入配置，不要硬编码到业务逻辑。
- task 入参使用稳定 schema，优先基本类型或 DTO 字典。
- worker 日志、metrics、trace context 需要与主服务口径一致。
- 对外行为仍以业务服务层为中心，不要让 router 直接堆大量队列细节。
