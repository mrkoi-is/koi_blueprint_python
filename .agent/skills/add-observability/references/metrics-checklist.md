# Metrics Checklist

## 基础目标

- 暴露 `/metrics`
- 保留 `/health` 与 `/ready` 的独立语义
- 为请求量、延迟、错误率提供基础 HTTP 指标
- 为关键业务路径补充自定义计数器 / 直方图 / Gauge

## 接入顺序

1. 确认主服务有统一 app factory。
2. 在启动阶段挂载 metrics setup，而不是在 router 内零散初始化。
3. 对默认 HTTP 指标做路径排除或归一化，避免高基数标签。
4. 只为关键业务事件增加自定义 metrics，例如：
   - 任务入队数
   - 第三方调用失败数
   - 缓存命中率
   - WebSocket 当前连接数

## 设计约束

- 不要把用户 ID、订单号等高基数字段直接做 label。
- 指标名称保持前缀统一，例如 `<service>_` 或领域前缀。
- `/metrics` 若对外暴露，需要配合网络隔离或入口策略。
- 指标接入后，至少保留一个可本地验证的抓取方式。

## 验证清单

- `curl /metrics` 有响应
- Prometheus 可以成功抓取
- 核心路由访问后，计数器与延迟指标有变化
- 错误请求会进入错误率相关指标
